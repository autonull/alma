## First Principles: What Can Steering Actually Do?

Let's be brutally honest about what activation steering can and cannot do:

### ✅ Steering CAN

| Capability | Evidence | Practical Value |
|------------|----------|-----------------|
| **Change style/register** | Turner et al. (2023): AAVE, formality, sentiment | Moderate — fine-tuning does this better |
| **Shift topic probability** | Intuition: steering toward "biology" embeddings | Low — prompting does this fine |
| **Reduce harmful outputs** | Some safety steering work | Moderate — RLHF does this better |
| **Inject simple behaviors** | Counting, formatting, structure | Low — prompting does this fine |

### ❌ Steering CANNOT

| Limitation | Why |
|------------|-----|
| **Add new knowledge** | Weights are frozen; model can't know what it didn't learn |
| **Improve reasoning** | GSM8K performance depends on weights, not activations |
| **Fix factual errors** | If model believes X, steering won't make it believe not-X |
| **Generalize to new domains** | Steering vectors are context-specific |

### The Uncomfortable Conclusion

> **For most practical tasks, fine-tuning or prompting outperforms steering.**

Steering is:
- Less powerful than fine-tuning (can't add knowledge)
- Less reliable than prompting (activation space is opaque)
- More complex than both (requires training a planner)

**So why build ALMA at all?**

## The Actual Value Proposition

There are exactly **three** problems where ALMA might have an advantage:

### 1. Rapid Adaptation Without Weight Changes

**Problem:** You need the model to behave differently *now*, not after hours of fine-tuning.

**Example:** A customer service bot needs to adapt its tone based on customer mood (detected in real-time).

**Why ALMA helps:**
- Steering is inference-time only — no weight updates
- Planner can switch steering vectors in milliseconds
- Fine-tuning takes minutes to hours; steering takes milliseconds

**Benchmark:** Customer satisfaction with adaptive vs. static responses.

---

### 2. Multi-Task Behavior with Single Model

**Problem:** You need one model to behave as 10 different "personalities" without loading 10 LoRA adapters.

**Example:** A game NPC that shifts between "friendly merchant," "hostile guard," "wise elder" based on context.

**Why ALMA helps:**
- One model + multiple steering vectors vs. one model + multiple LoRA weights
- Steering vectors are smaller (960-dim) than LoRA adapters (millions of params)
- Switching is cleaner (no weight loading/unloading)

**Benchmark:** Memory footprint, switch latency, behavior quality.

---

### 3. Interpretable Behavior Modification

**Problem:** You need to understand *why* the model changed its output.

**Example:** A medical assistant that must explain why it gave a particular recommendation.

**Why ALMA helps:**
- Steering vectors are inspectable (unlike fine-tuned weight changes)
- Planner decisions are logged (latent goal → steering vector → output)
- You can trace: "Output changed because steering vector X was applied"

**Benchmark:** Interpretability scores, audit trail completeness.

---

## What ALMA v4 Actually Is

**ALMA v4 is a tool for:**

> **Real-time, interpretable behavior modification of language models without weight updates.**

**Target users:**
- Developers building adaptive agents (tone, style, persona shifts)
- Researchers studying LLM behavior (mechanistic interpretability)
- Applications requiring audit trails (why did the model do this?)

**Not for:**
- Improving benchmark performance (GSM8K, MMLU, etc.)
- Adding new knowledge to models
- Tasks where fine-tuning is acceptable

## Core Design — Problem-Driven

### Use Case 1: Adaptive Customer Service

**Scenario:** Bot detects customer frustration → shifts to empathetic tone → de-escalates.

**Architecture:**
```
Customer message → Sentiment detector → Planner → Steering vector → Response
                                              ↓
                                    (empathy, patience, formality)
```

**Training:**
- Reward = customer satisfaction (explicit rating or inferred from sentiment improvement)
- Planner learns which steering vectors reduce frustration
- No fine-tuning needed; same base model handles all tones

**Benchmark:**
- Baseline: Static customer service model
- ALMA: Adaptive model with steering
- Metric: Avg customer satisfaction, resolution rate

---

### Use Case 2: Multi-Persona Game NPC

**Scenario:** NPC shifts between personas based on player interaction.

**Architecture:**
```
Player action → Context classifier → Planner → Steering vector → NPC response
                                              ↓
                                    (friendly/hostile/mysterious/etc.)
```

**Training:**
- Reward = player engagement (time spent, positive interactions)
- Planner learns persona transitions that maximize engagement
- Pre-trained steering vectors for each persona

**Benchmark:**
- Baseline: Single-persona NPC
- ALMA: Multi-persona NPC
- Metric: Player engagement time, satisfaction surveys

---

### Use Case 3: Interpretable Decision Support

**Scenario:** Model recommends actions with audit trail of *why*.

**Architecture:**
```
Query → Planner → Steering vector → Response + Explanation
                        ↓
              "Steering: factuality=0.8, caution=0.6"
```

**Training:**
- Reward = accuracy + human trust ratings
- Planner learns to apply steering vectors that increase trust
- All steering decisions logged for audit

**Benchmark:**
- Baseline: Standard model output
- ALMA: Model output + steering audit trail
- Metric: Human trust scores, decision accuracy

## Training Protocol — Clear and Specific

### Phase 1: Pre-train Steering Vectors (Offline)

**Goal:** Build a library of behavior steering vectors.

**Method:**
```python
for behavior in ["empathetic", "formal", "casual", "technical", "cautious"]:
    # Collect examples of this behavior
    examples = load_examples(behavior)
    
    # Find steering vector that maximizes behavior classifier score
    vector = optimize_steering_vector(
        model, examples, 
        reward_fn=behavior_classifier(behavior)
    )
    
    # Store in library
    steering_library[behavior] = vector
```

**Time:** ~1 hour per behavior (one-time cost)

**Output:** `steering_library.pt` with 10-20 behavior vectors

---

### Phase 2: Train Planner (Task-Specific)

**Goal:** Learn when and how to apply steering vectors for a specific task.

**Method:**
```python
for episode in range(num_episodes):
    # Get context (e.g., customer message)
    context = get_context()
    
    # Planner selects steering vector(s)
    steering_choice = planner(context)
    steering_vector = steering_library[steering_choice]
    
    # Generate response with steering
    response = model.generate(context, steering=steering_vector)
    
    # Compute reward (e.g., customer satisfaction)
    reward = compute_reward(response, context)
    
    # Update planner via PPO
    planner.update(reward)
```

**Time:** 500-1000 steps (~10-30 minutes on RTX 3060)

**Output:** `planner_taskname.pt`

---

### Phase 3: Deploy

**Goal:** Run trained planner in production.

**Method:**
```python
# Load
planner.load("planner_customer_service.pt")
steering_library.load("steering_library.pt")

# Run
while True:
    context = get_user_input()
    steering_choice = planner(context)
    steering_vector = steering_library[steering_choice]
    response = model.generate(context, steering=steering_vector)
    log(context, response, steering_choice)  # Audit trail
    yield response
```

**Latency:** +5-10ms over baseline (planner inference + ActAdd)

## What About Benchmarks?

**Honest answer:** ALMA probably won't help with GSM8K, MMLU, HellaSwag, etc.

**Why:**
- These benchmarks test *knowledge* and *reasoning*
- Steering doesn't add knowledge
- Fine-tuning or better prompting is the right tool

**Exception:** If a benchmark measures *behavior style* (e.g., "respond in Shakespearean English"), then steering could help. But that's not what most benchmarks measure.

**Alternative benchmark suite for ALMA:**

| Benchmark | What it measures | Target |
|-----------|------------------|--------|
| **Tone Adaptation** | Can it shift tone based on context? | Classifier accuracy >85% |
| **Persona Consistency** | Does it maintain a persona over 10+ turns? | Consistency score >0.8 |
| **Switch Latency** | How fast can it change behavior? | <10ms overhead |
| **Audit Completeness** | Can you trace why it did something? | 100% of decisions logged |
| **Memory Efficiency** | How much overhead vs. LoRA? | <1% of LoRA memory |

These measure what ALMA actually does, not what it doesn't do.

│  ┌──────────────────────────────────────────────────────────┐│
│  │              Reward Signal (Task-Specific)               ││
│  │  • Customer satisfaction                                 ││
│  │  • Player engagement                                     ││
│  │  • Human trust scores                                    ││
│  │  • Task success                                          ││
│  └──────────────────────────────────────────────────────────┘│
│                                                              │
│  ┌──────────────────────────────────────────────────────────┐│
│  │              Audit Log (Always On)                       ││
│  │  [timestamp] [context] [steering_choice] [output]        ││

**Lines of code:** ~600 (down from 800 in v3)

**Components removed:**
- ❌ Memory buffer for novelty (not needed for task-specific use)
- ❌ Intrinsic reward (novelty, coherence) — replaced with task reward
- ❌ Mode switching (flat/hierarchical) — not needed
- ❌ Model cascade — orthogonal concern

**Components added:**
- ✅ Steering library (pre-trained behavior vectors)
- ✅ Audit log (interpretability)
- ✅ Task-specific reward (not intrinsic curiosity)

## Three Concrete Applications

### Application 1: Customer Service Adapter

**What it does:** Detects customer frustration, adapts tone to de-escalate.

**Setup:**
```yaml
# configs/customer_service.yaml

steering_library:
  behaviors: ["empathetic", "patient", "apologetic", "solution-focused"]
  
planner:
  context_encoder: "sentiment_model"
  hidden_dim: 128
  
reward:
  type: "customer_satisfaction"
  source: "rating"  # or "sentiment_improvement"
  
training:
  steps: 500
  episodes_per_step: 4
  
deployment:
  log_all_decisions: true
  latency_budget: 50ms  # including steering
```

**Expected outcome:** +15-20% customer satisfaction vs. static bot.

## Honest Limitations

### What ALMA v4 Will NOT Do

| Claim | Reality |
|-------|---------|
| "Improve GSM8K scores" | No — steering doesn't add reasoning ability |
| "Make the model smarter" | No — it's the same frozen weights |
| "Replace fine-tuning" | No — different tool for different job |
| "Work out of the box" | No — requires task-specific planner training |
| "Be simpler than prompting" | No — more complex, but more powerful for specific use cases |

### When NOT to Use ALMA

- You need the model to know new facts → **Fine-tune or RAG**
- You need better benchmark performance → **Fine-tune**
- You need a quick solution → **Prompt engineering**
- You can't tolerate +5-10ms latency → **Don't use steering**
- You don't need interpretability → **Fine-tune or RLHF**

### When TO Use ALMA

- You need real-time behavior adaptation → **ALMA**
- You need multiple personas from one model → **ALMA**
- You need audit trails for decisions → **ALMA**
- You can't fine-tune (regulatory, deployment constraints) → **ALMA**
